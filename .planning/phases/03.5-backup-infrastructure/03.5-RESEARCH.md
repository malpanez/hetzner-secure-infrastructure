## Summary

Phase 3.5 provisions automated encrypted daily backups for both WordPress databases and media directories. Backups flow to a Hetzner Object Storage bucket using the `hashicorp/aws` Terraform provider pointed at the FSN1 S3-compatible endpoint. Credentials are never written to disk — the backup script reads them from OpenBao KV at runtime using the established `bao kv get -field` pattern already present in this codebase.

The three-plan structure (Terraform bucket, Ansible role, OpenBao integration) maps cleanly to three independent implementation units. Terraform provisions the bucket and lifecycle rules. The Ansible `backup` role installs the script, systemd service/timer, and aws-cli. The OpenBao integration adds the `secret/backup` KV seeding to the bootstrap task file and creates the `backup-reader` policy + token.

**Primary recommendation:** Implement in plan order — Terraform first (03.5-01), then Ansible role (03.5-02), then OpenBao bootstrap additions (03.5-03). The Terraform plan can be applied before the server rebuild; the Ansible role is applied during the server rebuild.

**Critical constraint:** aws-cli v2 has no apt package for Debian. The only supported install method is the official bundled installer downloaded from `awscli.amazonaws.com` for `aarch64`. This must be a task in the Ansible role, not an apt install.

### Pattern 1: Terraform — hashicorp/aws Provider for Hetzner S3

**What:** Use `hashicorp/aws` provider with `endpoints { s3 = "..." }` override to target Hetzner's S3-compatible endpoint. All AWS-validation checks must be skipped.

**When to use:** Any time Hetzner Object Storage needs to be managed via Terraform.

```hcl
# Source: https://gist.github.com/3deep5me/2571b3070fb6bc243b1664d60eb2d5d1
# verified 2026-03-29

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true
  skip_region_validation      = true

  endpoints {
    s3 = "https://nbg1.your-objectstorage.com"
  }

  region with default "nbg1"
  access_key = var.hetzner_s3_access_key
  secret_key = var.hetzner_s3_secret_key
}

resource "aws_s3_bucket" "tmt_backups" {
  bucket = "tmt-backups"
}

resource "aws_s3_bucket_acl" "tmt_backups" {
  bucket = aws_s3_bucket.tmt_backups.id
  acl    = "private"
}
```

**IMPORTANT:** The `hashicorp/aws` provider must be added as a new provider declaration alongside the existing `hcloud` and `cloudflare` providers in the root `terraform/main.tf` `required_providers` block. It will be a new provider alias or a new module. Because the root `main.tf` already has a `provider "aws"` block would conflict if placed at root level — use a **separate Terraform module** (`terraform/modules/object-storage/`) with its own `required_providers` block.

### Pattern 2: S3 Lifecycle Rules — Prefix-Based Retention

**What:** Three sets of objects with different prefixes carry different expiration rules. The backup script writes files with prefix paths encoding their retention class (e.g., `daily/`, `weekly/`, `monthly/`). S3 lifecycle rules then expire each prefix at the matching age.

**Retention strategy:**

- `daily/` prefix: expire after 7 days
- `weekly/` prefix: expire after 28 days (4 weeks)
- `monthly/` prefix: expire after 90 days (3 months)

The **script** decides which prefix to use based on day-of-week (Sunday = weekly) and day-of-month (1st = monthly), defaulting to `daily/`.

```hcl
# Source: Terraform Registry aws_s3_bucket_lifecycle_configuration + Hetzner lifecycle docs
# verified pattern 2026-03-29

resource "aws_s3_bucket_lifecycle_configuration" "tmt_backups" {
  bucket = aws_s3_bucket.tmt_backups.id

  rule {
    id     = "expire-daily"
    status = "Enabled"

    filter {
      prefix = "daily/"
    }

    expiration {
      days = 7
    }
  }

  rule {
    id     = "expire-weekly"
    status = "Enabled"

    filter {
      prefix = "weekly/"
    }

    expiration {
      days = 28
    }
  }

  rule {
    id     = "expire-monthly"
    status = "Enabled"

    filter {
      prefix = "monthly/"
    }

    expiration {
      days = 90
    }
  }
}
```

**Hetzner lifecycle note (MEDIUM confidence):** Hetzner Object Storage supports prefix-based lifecycle expiration via the S3 API. Lifecycle policies do NOT work if versioning is suspended — the bucket must have either no versioning (default) or versioning enabled (not suspended). Since the backup bucket does not need versioning, leave it disabled (default). Confirmed via Hetzner docs search 2026-03-29.

### Pattern 3: OpenBao Runtime Credential Injection

**What:** Script reads credentials from OpenBao KV at runtime using `bao kv get -field`. Token is read from `/root/.openbao-backup-token`. No credentials written to environment files or disk beyond the token file.

**This exact pattern is already used in the codebase** (`rotate-mariadb-root.sh`).

```bash
# Source: ansible/playbooks/setup-openbao-rotation.yml (existing codebase pattern)

export VAULT_TOKEN
VAULT_TOKEN=$(cat /root/.openbao-backup-token)
export VAULT_ADDR="http://127.0.0.1:8200"

bao token renew-self > /dev/null 2>&1 || true

S3_ACCESS_KEY=$(bao kv get -field=s3_access_key secret/backup)
S3_SECRET_KEY=$(bao kv get -field=s3_secret_key secret/backup)
ENCRYPT_PASS=$(bao kv get -field=encryption_passphrase secret/backup)
```

**Environment variable note:** OpenBao CLI accepts both `VAULT_TOKEN`/`VAULT_ADDR` (HashiCorp Vault compatibility) and `BAO_TOKEN`/`BAO_ADDR`. The existing codebase uses `VAULT_TOKEN`/`VAULT_ADDR` — use the same for consistency.

### Pattern 4: Encrypt-then-Upload (no plaintext staging)

**What:** Pipe the backup output through openssl before writing to disk or uploading. For database dumps, pipe mysqldump → gzip → openssl → file. For media, pipe tar → gzip → openssl → file. The encrypted file is the only artifact that touches disk (in `/var/backups/`), and it is deleted after upload.

```bash
# Source: OpenSSL docs + codebase pattern (openssl rand already used in rotate-mariadb-root.sh)
# AES-256-CBC with PBKDF2 (required; default digest sha-256)
# -pass stdin avoids passphrase appearing in process list or command history

TIMESTAMP=$(date +%Y%m%d-%H%M%S)
BACKUP_FILE="/var/backups/wordpress_main-${TIMESTAMP}.sql.gz.enc"

mysqldump \
  --single-transaction \
  --skip-lock-tables \
  --routines \
  --events \
  -u root -p"${MARIADB_ROOT_PASS}" \
  wordpress_main \
  | gzip \
  | openssl enc -aes-256-cbc -salt -pbkdf2 \
    -pass "pass:${ENCRYPT_PASS}" \
    -out "${BACKUP_FILE}"

# Upload with endpoint override
AWS_ACCESS_KEY_ID="${S3_ACCESS_KEY}" \
AWS_SECRET_ACCESS_KEY="${S3_SECRET_KEY}" \
  aws s3 cp "${BACKUP_FILE}" \
    "s3://tmt-backups/${BACKUP_PREFIX}/wordpress_main-${TIMESTAMP}.sql.gz.enc" \
    --endpoint-url "https://nbg1.your-objectstorage.com"

rm -f "${BACKUP_FILE}"
```

**Note on S3 credentials:** Pass `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` as environment variables to `aws s3 cp`. Do NOT create a persistent `~/.aws/credentials` file — that would write secrets to disk.

### Pattern 5: Post-Upload Verification

**What:** After upload, use `aws s3api head-object` to confirm the object exists and has non-zero ContentLength.

```bash
SIZE=$(AWS_ACCESS_KEY_ID="${S3_ACCESS_KEY}" \
  AWS_SECRET_ACCESS_KEY="${S3_SECRET_KEY}" \
  aws s3api head-object \
    --bucket tmt-backups \
    --key "${BACKUP_PREFIX}/wordpress_main-${TIMESTAMP}.sql.gz.enc" \
    --endpoint-url "https://nbg1.your-objectstorage.com" \
    --query ContentLength \
    --output text 2>/dev/null)

if [[ -z "${SIZE}" || "${SIZE}" -eq 0 ]]; then
  log "ERROR: Upload verification failed for wordpress_main backup"
  exit 1
fi
log "Upload verified: ${SIZE} bytes"
```

### Anti-Patterns to Avoid

- **`apt install awscli`:** Installs aws-cli v1 on Debian, not v2. Always use the bundled installer.
- **Storing S3 credentials in `~/.aws/credentials`:** This writes secrets to disk. Use environment variables per invocation.
- **Using `--lock-tables` with InnoDB mysqldump:** Incompatible with `--single-transaction`; degrades to a locking backup. Always use `--single-transaction --skip-lock-tables`.
- **Staging plaintext backups to disk:** Always pipe mysqldump/tar output through openssl. The only file written to disk is the encrypted `.enc` file.
- **Putting passphrase in `openssl -k` CLI flag:** The passphrase appears in `ps aux`. Use `-pass "pass:${VAR}"` where `$VAR` is a shell variable, or `-pass stdin`.
- **Using `provider "aws" {}` at root module level:** Would conflict with existing root providers (`hcloud`, `cloudflare`). Put the object storage Terraform in a separate `modules/object-storage/` module with its own provider block.
- **Using `aws_s3_bucket_object_lock_configuration`:** Object Lock requires bucket creation with `object_lock_enabled = true`; it is incompatible with non-versioned buckets and adds unnecessary complexity. Use simple lifecycle expiration instead.
- **Lifecycle rules with versioning suspended:** Hetzner lifecycle rules do not trigger when versioning is suspended. Leave versioning off entirely (default) for this bucket.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Backup retention enforcement | Custom cron job that deletes old files | S3 lifecycle rules | Atomic, server-side, no script failure risk; lifecycle rules are idempotent |
| Credential file management | Per-profile `~/.aws/credentials` | env vars per invocation (`AWS_ACCESS_KEY_ID=... aws s3 cp`) | No secrets on disk; matches existing codebase pattern |
| Key derivation for encryption | Raw AES key | `openssl enc -pbkdf2` | PBKDF2 protects against brute-force on passphrase; without it, openssl uses a deprecated MD5-based KDF |
| Backup scheduling | Custom Python/bash daemon | systemd timer | Already used in this project; `Persistent=true` handles missed runs |

## Common Pitfalls

### Pitfall 1: aws-cli v1 installed via apt

**What goes wrong:** `apt install awscli` on Debian 13 installs version 1.x which has different command syntax and limited S3-compatible endpoint support.
**Why it happens:** The Debian package is aws-cli v1, not v2.
**How to avoid:** Use the bundled installer from `awscli.amazonaws.com/awscli-exe-linux-aarch64.zip` in the Ansible task.
**Warning signs:** `aws --version` returns `aws-cli/1.x` — the task must assert `aws-cli/2.` prefix.

### Pitfall 2: Terraform provider conflict at root level

**What goes wrong:** Adding `provider "aws" {}` at `terraform/main.tf` root level conflicts with the existing `cloudflare` and `hcloud` providers if aliasing is not used correctly, or causes unexpected plan behavior.
**Why it happens:** The root module already has a comprehensive provider setup. Adding a second provider for a separate concern (object storage) at root level pollutes the state.
**How to avoid:** Create `terraform/modules/object-storage/` as a standalone module with its own `terraform { required_providers { aws = ... } }` block. Call it from root `main.tf` as `module "object_storage" { source = "./modules/object-storage" ... }`.

### Pitfall 3: OpenBao sealed at backup time

**What goes wrong:** Backup runs at 02:30 daily. If OpenBao is sealed (e.g., after an unplanned reboot), `bao kv get` returns exit code 2 and the script fails to retrieve credentials.
**Why it happens:** OpenBao requires manual unseal of the transit instance (8201) after every reboot — documented in CLAUDE.md.
**How to avoid:** Add an explicit OpenBao status check at the start of the backup script with a clear error message (matching the pattern in `rotate-mariadb-root.sh`). The failure will show in `journalctl -u backup.service`.
**Warning signs:** Script exits on `bao kv get` with `Error making API request: Code: 503`.

### Pitfall 4: Passphrase in process list

**What goes wrong:** If `openssl enc -k "${ENCRYPT_PASS}"` or `-pass pass:"${ENCRYPT_PASS}"` is used and the variable contains special characters, or if command substitution expands the variable in a logged context, the passphrase may appear in system logs.
**Why it happens:** `-k` flag is deprecated and insecure. Even `-pass pass:$VAR` can leak if the variable is logged.
**How to avoid:** Use `no_log: true` in any Ansible task that handles the passphrase. In the bash script, never `log` the passphrase variable. Use `set +x` before assignments that involve secrets.

### Pitfall 5: S3 lifecycle rules and Hetzner versioning interaction

**What goes wrong:** Lifecycle expiration rules do not delete objects if bucket versioning is "Suspended" — objects get delete markers but underlying versions remain.
**Why it happens:** S3 lifecycle rules for versioned buckets operate differently than for non-versioned buckets. Suspending versioning after enabling it leaves existing versions unmanaged.
**How to avoid:** Do NOT enable versioning on the `tmt-backups` bucket. Use the default state (versioning off). Lifecycle `expiration.days` will then cleanly delete objects.

### Pitfall 6: mysqldump MariaDB root password on CLI

**What goes wrong:** `-p"${MARIADB_ROOT_PASS}"` in mysqldump may be visible in process list.
**Why it happens:** CLI arguments are visible in `/proc/<pid>/cmdline` briefly.
**How to avoid:** Use a MySQL defaults file approach with a temp file, OR use `--defaults-extra-file` pattern, OR read the password from OpenBao directly via `MYSQL_PWD` environment variable. MYSQL_PWD is deprecated but functional and avoids the CLI arg; alternatively read from OpenBao into a variable. The existing codebase pattern uses `VAULT_TOKEN` variable injection — apply the same discipline.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `openssl enc` without `-pbkdf2` | `openssl enc -pbkdf2` | OpenSSL 1.1.1+ | Without `-pbkdf2`, OpenSSL uses deprecated EVP_BytesToKey with MD5; with `-pbkdf2` uses PBKDF2-HMAC-SHA256 |
| `aws_s3_bucket` with inline `lifecycle_rule` | Separate `aws_s3_bucket_lifecycle_configuration` resource | Terraform AWS provider v4+ | Inline lifecycle_rule deprecated in v4, removed in v5 — use standalone resource |
| `VAULT_TOKEN` env var only | Both `BAO_TOKEN` and `VAULT_TOKEN` accepted | OpenBao 1.x+ | OpenBao maintains backward compat with VAULT_*vars; use VAULT_* to match existing codebase |

**Deprecated/outdated:**

- `openssl enc -k password` — deprecated flag; use `-pass pass:$VAR` or `-pass stdin`
- `aws_s3_bucket.lifecycle_rule` inline block — removed in hashicorp/aws v5+; use `aws_s3_bucket_lifecycle_configuration`
- `apt install awscli` — installs v1, not v2; unsupported install path for this project

## Open Questions

1. **Hetzner S3 access key scoping**
   - What we know: Hetzner S3 keys are project-scoped — a single key pair grants read/write access to ALL buckets in the same Hetzner project. There is no per-bucket scoping via IAM-equivalent policies.
   - What's unclear: Whether the Terraform `aws_s3_bucket_acl` resource with `acl = "private"` behaves identically to standard AWS private ACL, or whether it is effectively a no-op since Hetzner keys are project-scoped.
   - Recommendation: Create the bucket with `acl = "private"` as belt-and-suspenders. The credentials stored in OpenBao are the primary access control.

2. **aws-cli v2 `--endpoint-url` for aws s3api head-object**
   - What we know: `aws s3 cp` accepts `--endpoint-url`. `aws s3api head-object` also accepts `--endpoint-url`.
