**tasks/install.yml** (BACKUP-02 dependency: aws-cli + mariadb-client):
1. Install `mariadb-client`, `jq`, `unzip` via `ansible.builtin.apt` with `state: present` and `update_cache: true`.
2. Check if aws-cli v2 is already installed: `ansible.builtin.command: /usr/local/bin/aws --version`, register `aws_cli_check`, `changed_when: false`, `failed_when: false`.
3. Download aws-cli v2 bundled installer: `ansible.builtin.get_url` with `url: "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip"`, `dest: /tmp/awscliv2.zip`, `mode: '0644'`, `when: aws_cli_check.rc != 0`.
4. Unzip: `ansible.builtin.unarchive` with `src: /tmp/awscliv2.zip`, `dest: /tmp/`, `remote_src: true`, `creates: /tmp/aws/install`, `when: aws_cli_check.rc != 0`.
5. Install: `ansible.builtin.command` with `cmd: /tmp/aws/install`, `creates: /usr/local/bin/aws`, `when: aws_cli_check.rc != 0`.
6. Clean up: `ansible.builtin.file` with `path: "{{ item }}"`, `state: absent` for `/tmp/awscliv2.zip` and `/tmp/aws`, `when: aws_cli_check.rc != 0`.
7. Verify aws-cli v2: `ansible.builtin.command: /usr/local/bin/aws --version`, register `aws_version_check`, `changed_when: false`.
8. Assert aws-cli v2: `ansible.builtin.assert` that `"'aws-cli/2' in aws_version_check.stdout"`.

**templates/backup.service.j2:**
```ini
[Unit]
Description=Encrypted backup of WordPress databases and media to Hetzner Object Storage
After=network.target openbao.service mariadb.service
Requires=openbao.service mariadb.service

[Service]
Type=oneshot
User=root
ExecStart=/usr/local/bin/backup.sh
StandardOutput=journal
StandardError=journal
SyslogIdentifier=backup-wordpress

[Install]
WantedBy=multi-user.target
```

**templates/backup.timer.j2:**

```ini
[Unit]

OnCalendar=*-*-* 02:30:00
Persistent=true
RandomizedDelaySec=60

[Install]
WantedBy=timers.target
```

**templates/backup.sh.j2** — This is the core script. Structure it as follows:

1. Shebang `#!/bin/bash` and `set -euo pipefail`.
2. Configuration section with variables from Jinja: `OPENBAO_ADDR="{{ backup_openbao_addr }}"`, `TOKEN_FILE="{{ backup_token_file }}"`, `LOG_FILE="{{ backup_log_file }}"`, `STAGING_DIR="{{ backup_staging_dir }}"`, `S3_BUCKET="{{ backup_bucket }}"`, `S3_ENDPOINT="{{ backup_s3_endpoint }}"`.
3. `log()` and `error_exit()` functions (same pattern as `rotate-mariadb-root.sh`).
4. OpenBao status check: `if ! systemctl is-active --quiet openbao; then error_exit "OpenBao is not running"; fi`.
5. Token file check and `export VAULT_TOKEN` / `export VAULT_ADDR` (using `VAULT_TOKEN` and `VAULT_ADDR` for compatibility, matching existing codebase pattern).
6. Token self-renewal: `bao token renew-self > /dev/null 2>&1 || log "WARN: Token renewal failed, continuing"`.
7. Read credentials from OpenBao:
   ```
   S3_ACCESS_KEY=$(bao kv get -field=s3_access_key secret/backup)
   S3_SECRET_KEY=$(bao kv get -field=s3_secret_key secret/backup)
   ENCRYPT_PASS=$(bao kv get -field=encryption_passphrase secret/backup)
   MARIADB_ROOT_PASS=$(bao kv get -field=root_password secret/mariadb)
   ```
8. Backup prefix logic: `DOW=$(date +%u)`, `DOM=$(date +%d)`. If `DOM == "01"` then `BACKUP_PREFIX="monthly"`, elif `DOW == "7"` then `BACKUP_PREFIX="weekly"`, else `BACKUP_PREFIX="daily"`.
9. Timestamp: `TIMESTAMP=$(date +%Y%m%d-%H%M%S)`.
10. `upload_and_verify()` function that takes a local file path and S3 key as arguments:
    - Uploads with `AWS_ACCESS_KEY_ID="${S3_ACCESS_KEY}" AWS_SECRET_ACCESS_KEY="${S3_SECRET_KEY}" aws s3 cp "$1" "s3://${S3_BUCKET}/$2" --endpoint-url "${S3_ENDPOINT}"`.
    - Verifies with `aws s3api head-object --bucket "${S3_BUCKET}" --key "$2" --endpoint-url "${S3_ENDPOINT}" --query ContentLength --output text`. Check size > 0. Log verified size. Exit 1 on failure.
    - Both aws commands use inline `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` env vars (never `~/.aws/credentials`).
11. Database backup loop (BACKUP-02): iterate over sites defined in a Jinja loop:
    ```
    {% raw %}
    {% for site in backup_sites %}
    ```
    For each site: `mariadb-dump --single-transaction --skip-lock-tables --routines --events -u root -p"${MARIADB_ROOT_PASS}" {{ site.db_name }} | gzip | openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000 -pass "pass:${ENCRYPT_PASS}" -out "${STAGING_DIR}/{{ site.db_name }}-${TIMESTAMP}.sql.gz.enc"`. Use `MYSQL_PWD="${MARIADB_ROOT_PASS}" mariadb-dump` pattern instead of `-p` flag (avoids password in process list). Then call `upload_and_verify` with the local file and `"${BACKUP_PREFIX}/{{ site.db_name }}-${TIMESTAMP}.sql.gz.enc"`. Then `rm -f` the local file.
    ```
    {% endfor %}
    {% endraw %}
    ```
12. Media backup loop (BACKUP-03): same Jinja loop. For each site: `tar -czf - -C {{ site.web_root }}/wp-content uploads | openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000 -pass "pass:${ENCRYPT_PASS}" -out "${STAGING_DIR}/{{ site.name }}-media-${TIMESTAMP}.tar.gz.enc"`. Upload, verify, remove.
13. Final log line with total backup count.

Use `mariadb-dump` as the command (Debian 13 canonical name). Add a fallback check at the top of the script: `DUMP_CMD=$(command -v mariadb-dump || command -v mysqldump)` and use `${DUMP_CMD}` throughout. This handles both binary names.

IMPORTANT: For the `MYSQL_PWD` pattern — set `export MYSQL_PWD="${MARIADB_ROOT_PASS}"` before the dump loop and `unset MYSQL_PWD` after. This avoids the password appearing in process args via `-p`. The dump command becomes just `${DUMP_CMD} --single-transaction --skip-lock-tables --routines --events -u root {{ site.db_name }}`.
