# Phase 3.6 Plan 02: Monitoring Ports Binding Summary

Bind Prometheus (9090), Loki gRPC (9096), and Node Exporter (9100) to 127.0.0.1 on prod-de-wp-01. Add UFW deny rules for defense-in-depth.

## What Was Done

### Task 1: Discovery

Examined the monitoring role structure. Found it only manages rsyslog/security scripts — not the observability stack. The actual services (Prometheus, Loki, Node Exporter) are managed by external collections (`prometheus.prometheus`, `grafana.grafana`) with variables in `group_vars/monitoring_servers/`.

Confirmed on server: ports 9090, 9096, 9100 all bound to `*` (all interfaces). Port 3100 (Loki HTTP) already bound to `127.0.0.1`.

### Task 2: Variable fixes (Ansible group_vars)

- `prometheus.yml`: `prometheus_listen_address: "0.0.0.0"` -> `"127.0.0.1"`
- `node_exporter.yml`: `node_exporter_listen_address: "0.0.0.0"` -> `"127.0.0.1"`
- `loki.yml`: added `grpc_listen_address: "127.0.0.1"` to `loki_server` block

### Task 3: UFW deny rules

Added to `ansible/roles/firewall/tasks/rules.yml` before the "Verify SSH rule exists" task:

```yaml
- name: Firewall | Rules | Deny external access to monitoring ports (C3)
  community.general.ufw:
    rule: deny
    port: "{{ item }}"
    proto: tcp
  loop: ["9090", "9096", "9100"]
  tags: [firewall, rules, monitoring]
```

### Task 4: Applied to server

Applied directly via SSH (see deviation below):

- Patched `/etc/systemd/system/prometheus.service`: `--web.listen-address=0.0.0.0:9090` -> `127.0.0.1:9090`
- Patched `/etc/systemd/system/node_exporter.service`: `--web.listen-address=0.0.0.0:9100` -> `127.0.0.1:9100`
- Patched `/etc/loki/config.yml`: added `grpc_listen_address: 127.0.0.1`
- Applied UFW rules: `ufw deny 9090/tcp && ufw deny 9096/tcp && ufw deny 9100/tcp`
- Reloaded systemd and restarted all three services

### Task 5: Verification


All monitoring ports confirmed localhost-only. UFW deny rules active.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Applied via SSH instead of Ansible playbook**

- **Found during:** Task 4
- **Issue:** `wordpress-prod` is in `monitored_servers` group, not `monitoring_servers`. The monitoring playbook targets `monitoring_servers` only. Running the playbook against `--limit wordpress-prod --tags monitoring` would have applied the custom monitoring role (rsyslog) not the collection-based Prometheus/Loki/Node Exporter services.
- **Fix:** Patched systemd service files and loki config directly via SSH for immediate effect. The group_vars changes in Ansible ensure the next full collection-based deploy will be idempotent.
- **Files modified:** `/etc/systemd/system/prometheus.service`, `/etc/systemd/system/node_exporter.service`, `/etc/loki/config.yml` (on server)
