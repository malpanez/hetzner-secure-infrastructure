# Phase 3.6 Plan 03: PHP Hardening + AppArmor Enforce Mode Summary

PHP-FPM hardening (disable_functions, allow_url_fopen, request_terminate_timeout) applied to both WordPress pool files, and AppArmor switched from complain to enforce mode for php-fpm, sshd, and fail2ban profiles.

## What Was Done

### Task 1: AppArmor Complain Log Check (H1)

- Ran `aa-status` and checked `journalctl` for denied entries over 24 hours
- Result: zero AppArmor denied entries — clean log confirmed safe to enforce
- Updated `apparmor_enforce_mode: false` to `apparmor_enforce_mode: true` in `group_vars/hetzner/security.yml`

### Task 2: PHP-FPM Hardening (H2, M2)

- Identified that `php-fpm-wordpress.conf.j2` is the active template (not `php-fpm-pool.conf.j2`)
- Added `request_terminate_timeout` using dedicated variable `nginx_wordpress_php_fpm_request_terminate_timeout`
- Added `php_admin_value[disable_functions]` with reduced attack surface function list
- Added `php_admin_flag[allow_url_fopen] = off` (using correct boolean flag type)
- Added `nginx_wordpress_php_fpm_request_terminate_timeout: "300s"` to defaults/main.yml
- Both templates updated for consistency

### Task 3: Apply to Server

- Ran `ansible-playbook playbooks/dual-wordpress.yml` to deploy PHP-FPM pool changes (changed=2)
- Ran `ansible-playbook ... --tags apparmor` to apply enforce mode (changed=1)
- Both `main.conf` and `academy.conf` pool files updated with security settings
- `php-fpm` AppArmor profile enforced successfully

### Task 4: Verification

- `sudo aa-status`: 21 profiles in enforce mode (was 14), php-fpm and sshd both enforced
- Both pool files contain disable_functions, allow_url_fopen off, request_terminate_timeout
- `php8.4-fpm` service active with no errors

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Fix AppArmor local stub files for enforcement**
- **Found during:** Task 3 (applying AppArmor enforce mode)
- **Issue:** `aa-enforce` failed with "Include file /etc/apparmor.d/local/usr.bin.fail2ban-server not found" because system AppArmor profiles reference local include stubs that didn't exist, blocking enforcement
- **Fix:** Added task to `apparmor/tasks/configure.yml` to create stub files in `apparmor_local_dir` for managed profiles before enforcement
- **Files modified:** `ansible/roles/apparmor/tasks/configure.yml`
- **Commit:** 2c0dc54

**2. [Rule 1 - Bug] Fix apparmor_profiles to use profile filenames not binary paths**
- **Found during:** Task 3 (investigating why enforce wasn't working)
- **Issue:** `apparmor_profiles` had `/usr/sbin/sshd` and `/usr/bin/fail2ban-server` (binary paths), but the enforce task runs `aa-enforce /etc/apparmor.d/{{ item }}` which needs profile filenames
- **Fix:** Changed to `usr.sbin.sshd`, `usr.bin.fail2ban-server`, `php-fpm`
- **Files modified:** `ansible/inventory/group_vars/hetzner/security.yml`
- **Commit:** 2c0dc54

**3. [Rule 1 - Bug] Wrong template file — php-fpm-pool.conf.j2 is unused**
- **Found during:** Task 2 (verifying pool changes didn't apply to server)
- **Issue:** Plan referenced `php-fpm-pool.conf.j2` but the role deploys `php-fpm-wordpress.conf.j2`
- **Fix:** Applied hardening to `php-fpm-wordpress.conf.j2` (the active template); also updated `php-fpm-pool.conf.j2` for consistency
- **Files modified:** `ansible/roles/nginx_wordpress/templates/php-fpm-wordpress.conf.j2`
- **Commit:** 2c0dc54

**4. [Rule 2 - Missing Critical] allow_url_fopen should be php_admin_flag not php_admin_value**
- **Found during:** Task 2 (reviewing existing template content)
- **Issue:** `allow_url_fopen` is a boolean INI flag and must use `php_admin_flag` not `php_admin_value`
- **Fix:** Changed to `php_admin_flag[allow_url_fopen] = off`
- **Files modified:** Both PHP-FPM pool templates
- **Commit:** 2c0dc54

## Decisions Made

- `php-fpm-pool.conf.j2` is an unused legacy template — kept but not the one deployed by the role
- The `apparmor_profiles` variable now lists profile filenames (`usr.sbin.sshd`) to match the `aa-enforce /etc/apparmor.d/{{ item }}` command pattern
- Local AppArmor stub files (`/etc/apparmor.d/local/usr.sbin.sshd`, etc.) are required for enforcement; created by the role as empty stubs
- `request_terminate_timeout` gets its own dedicated role variable rather than reusing `php_max_execution_time`
